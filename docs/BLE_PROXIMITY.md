# BLE 近接（Proximity）

> 引き継ぎ索引: [HANDBOOK.md](./HANDBOOK.md) / コード: `lib/proximity/`

## 合成の考え方（BLE 優先ではない）

各端末は **GPS 距離（鬼まで）** と **BLE スキャン** を `HybridProximityService` でマージします。

- 採用するのは「より脅威が高い帯」（`none` < `far` < `near` < `contact`）
- GPS だけでは `contact` にならない（最大 `near`）
- **BLE の `contact` は、同一ルームかつ「鬼としてアドバタイズ中」の端末だけ**を対象（他の逃走者の端末は無視）

捕獲は **逃走者端末**で判定します。

1. 拘束中（タッチロック / 捕獲結界）
2. かつ **BLE `contact`**（鬼が BLE オンで近い）**または GPS で約 12m 以内**（鬼位置は Firestore 同期）

→ **片方だけ BLE オフでも、GPS 経路でゲームは進行できます。**

## 両方オン

- 鬼端末が「鬼ビーコン」でアドバタイズ、逃走者がスキャン → 至近で BLE `contact`
- ルーム外・無関係な BLE は無視

## 両方オフ

- BLE は使わない。GPS + 鬼位置イベントのみ。
- 拘束後は GPS 近距離で捕獲可能。

## 片方だけオン

| 端末 | 挙動 |
|------|------|
| **BLE オン** | 鬼がオンなら BLE `contact` も使える。鬼がオフならスキャンは当たらず **GPS のみ**（不安定になりにくい）。 |
| **BLE オフ** | 常に GPS のみ。鬼の位置は Firestore の `hunter_position` を使用。 |

「オン側の判定を優先してオフ側が不利」という設計ではありません。  
オフ側は GPS、オン側は **GPS +（条件が揃えば）BLE** の足し算です。

### 味としての不安定さ

- GPS の誤差（ビル街・密な場所）
- BLE オン同士でも、端末や OS によってスキャン間隔は 4 秒前後
- 鬼が BLE オフのとき、オン側は BLE では拾えず GPS 頼み → やや「探す」感じになる

意図的なランダム要素として許容する想定です。

## 補足

- 権限・省電力で BLE が止まることがあります
- 必須機能ではありません（全員オフでもオンラインマルチは成立）
